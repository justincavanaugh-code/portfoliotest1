Enterprise Vulnerability Triage

I used the SIEM dashboard to conduct a deep-dive analysis of the three agent hosts. After the successful deployment of Wazuh, I immediately began receiving detailed information on my network. At first I was alarmed at how many notifications I received. However, using AI as a learning partner to augment my cybersecurity certifications and training, I quickly began to learn the difference between theoretical analysis and what takes place when working with machines generating actual data - especially in my own network. My goal was to move beyond "scan and patch" and into professional risk categorization.

CVE Pattern Analysis

I began by trianging and prioritizing vulnerabilities. I discovered that the majority were kernel-level vulnerabilities published within 14 days of the assessment, representing the "Patching Gap" where official vendor updates were not yet available.

The Triage Framework

  • Vendor-Pending: Acknowledged vulnerabilities awaiting official patch release.
  • Deferred/Ignored: Older vulnerabilities deferred by the vendor due to stability risks.
  • Targeted Remediation: Specific fixes requiring manual installation outside of general upgrades.

Threat Hunting & Alert Analysis

Using Wazuh's Threat Hunting module, I monitored real-time event streams across all agents. The dashboard surfaced alert groups spanning syslog, authentication events, rule triggers, configuration change detections, and systemd activity. Events were correlated against PCI DSS requirements, giving each alert a compliance context alongside its technical one.

The Vulnerability Detection module scanned all enrolled agents and surfaced a combined picture of the environment's CVE exposure. Across the lab, a variety of potential vulnerabilities were identified. The practical challenge was not just finding them, but reasoning about what to do with each category.

I quickly learned that a raw vulnerability count means little without context. The most important insight from this project was that a high number of "High" findings is not automatically a sign of poor maintenance — it often reflects the Patching Gap: the window between when a CVE is published and when a vendor releases an official fix. Learning this distinction was fundamental to my objective of creating a simulated cybersecurity environment where I could apply real-life concepts.

Technical Challenges

This was actually the second time I had installed the Wazuh SIEM dashboard within my network. Initially, I'd deployed Wazuh using a Linux host machine utilizing KVM-based virtualization. I was able to login and even begin gathering data on my network. However, I eventually realized that VMWare on my Windows host was a more stable option in this case. Although this was an obstacle at first, it gave me a reason to gain additional experience with virtualization on a Windows host and VMWare. Previously, I'd almost exclusively used a Linux host/KVM for this type of project.

Technical Implementation

The deployment followed a containerized architecture to ensure scalability and isolation of security services.

Infrastructure as Code

Leveraged Docker Compose to orchestrate the multi-container environment, managing service dependencies and persistent data volumes to ensure configuration stayed intact across reboots.

Security & Encryption

Navigated version-specific certificate requirements by using the indexer-certs-creator tool to generate custom RSA certificates and private keys, replacing default insecure configurations for secure SSL/TLS communication.

Network Engineering

Configured Docker bridge networking and specific port mappings to allow secure remote access from the host browser to the virtualized internal dashboard.

Outcomes & Skills

  • Critical Triage: Demonstrated the ability to distinguish between actionable risks and "acceptable" risks based on vendor patch availability.
  • SIEM Proficiency: Gained realistic experience in inventory management, SCA (Security Configuration Assessment), and agent-to-manager troubleshooting.
  • Risk Management: Shifted focus to Compensating Controls (SCA hardening) when official software fixes were unavailable.
  • Using AI as a partner: Used multiple AI models to help me learn and navigate some of the more technical areas I was not familiar with going into this project. With AI's assistance, I was able to learn and complete the project much more quickly compared to working strictly from technical documentation.

Final Results & Summary

Established a fully functional SIEM dashboard in my home environment, generating valuable ongoing cybersecurity experience, as well as a more secure home network.

This project was designed to mirror what an SOC analyst actually does on the job: not just install a tool, but use it to monitor a real (if small) network, hunt for threats, assess compliance posture, and reason carefully about risk. By spanning three different endpoint types — a Windows workstation, a Linux desktop, and an Ubuntu virtual server — my SIEM dashboard generated the kind of varied, realistic security data that helped me further develop my understanding of cybersecurity.

The most lasting takeaway was operational: I learned that vulnerability management is not a "zero-out game". I gained a taste of what it's like to formulate an accurate, defensible picture of risk — and that requires knowing the difference between a vulnerability you can fix today, one you must wait on, and even some risk you may choose to accept. From installing a SIEM dashboard on a virtual server to monitoring my network and actually implementing security patches, this was a valuable (and ongoing) experience.
