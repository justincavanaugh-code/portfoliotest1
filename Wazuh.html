Deployment of a SIEM Dashboard on My Virtual Network

Role: Security Engineer / SOC Analyst (Home Lab)

Tools: Wazuh Docker Compose Ubuntu Server Virtualization OpenSSL

Project Overview

Successfully deployed a full-stack Wazuh dashboard (SIEM/XDR) environment within a virtualized Ubuntu instance on a Windows host. This project involved configuring a containerized security architecture including the Wazuh Indexer, Manager, and Dashboard. Beyond deployment, I utilized the environment to perform comprehensive vulnerability assessments and triage across a multi-platform lab (Windows 10, Ubuntu, and Linux Mint).

Technical Implementation

The deployment followed a containerized architecture to ensure scalability and isolation of security services.

Infrastructure as Code

Leveraged Docker Compose to orchestrate the multi-container environment, managing service dependencies and persistent data volumes to ensure configuration stayed intact across reboots.

Security & Encryption

Navigated version-specific certificate requirements by using the indexer-certs-creator tool to generate custom RSA certificates and private keys, replacing default insecure configurations for secure SSL/TLS communication.

Network Engineering

Configured Docker bridge networking and specific port mappings to allow secure remote access from the host browser to the virtualized internal dashboard.

Enterprise Vulnerability Triage

After deployment, I conducted a deep-dive analysis of a Linux-based server (Ubuntu 24.04). The goal was to move beyond "scan and patch" and into professional risk categorization.

CVE Pattern Analysis

Began by trianging and prioritizing vulnerabilities. I discovered that the majority were kernel-level vulnerabilities published within 14 days of the assessment, representing the "Patching Gap" where official vendor updates were not yet available.

The Triage Framework

  • Vendor-Pending: Acknowledged vulnerabilities awaiting official patch release.
  • Deferred/Ignored: Older vulnerabilities deferred by the vendor due to stability risks.
  • Targeted Remediation: Specific fixes requiring manual installation outside of general upgrades.

Key Challenge

The Hurdle: Encountered a v4.9.0-specific bug where certificate generation scripts created empty directories instead of valid .pem files, alongside Linux file ownership conflicts (UID 10001) that prevented the Docker daemon from reading keys.

The Fix: Used low-level diagnostic tools (ss, curl, docker inspect) to identify the root cause. I corrected the directory structures manually and adjusted permission sets to ensure the Wazuh services could initialize successfully.

Outcomes & Skills

  • Critical Triage: Demonstrated the ability to distinguish between actionable risks and "acceptable" risks based on vendor patch availability.
  • SIEM Proficiency: Gained advanced experience in Wazuh for inventory management, SCA (Security Configuration Assessment), and agent-to-manager troubleshooting.
  • Risk Management: Shifted focus to Compensating Controls (SCA hardening) when official software fixes were unavailable.
  • Using AI as a partner: Used multiple AI models to help me learn and navigate some of the more technical areas I was not familiar with going into this project. With AI's assistance, I was able to learn and complete the project much more quickly compared to working strictly from technical documentation.

Final Result: Established a fully functional, hardened SIEM environment with visibility into 200+ initial security alerts, providing immediate actionable data for ongoing system hardening.
