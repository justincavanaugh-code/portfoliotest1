AI-Assisted Gamified eLearning Demo

Inspired by ATD International Conference & EXPO 2026 (ATD26)

Role: Instructional Designer & Developer

Tools: HTML / CSS / JavaScript AI-Assisted (Vibe-Coding) GitHub Pages Agile Iteration

Project Overview

I was inspired by a session on vibe-coding at the 2026 Association for Talent Development Conference ("ATD26"). First, I was pleased to discover that I had already been "vibe coding": collaborating with AI to generate and refine code, and then customizing the code to generate output. But the session's live demonstration also taught me about several capabilities I had not considered. For example, the demonstration showed a fully functional gamified learning object being built and published in real time - with AI assistance — with no dedicated authoring tool, no SCORM package, and no LMS. The session challenged me to rethink how digital learning experiences can be created and distributed.

I left the session with a clear goal: expand my use of vibe-coding to replicate the demonstration, both in my home lab and in my work. I also used this project as an opportunity to include several concepts that were major themes of the overall ATD conference including: gamification, individualized learning experiences and cohort-inspired learning. The result was a series of progressively enhanced proof-of-concepts, each building on the last, and culminating in a gamified learning experience deployable from a simple website.

Note on content: The subject matter within the demo is intentionally generic (trivia-style questions). The goal was to demonstrate structure and mechanics without using proprietary or company-specific content.

Core Learning Design Concepts

Several design principles — drawn directly from session themes at the ATD26 conference — anchor every iteration of this project:

🏆

Gamification via Dynamic Leaderboard

Learners see their real-time rank among "peers", turning a solo activity into a friendly competition that sustains engagement and motivates completion.

👥

Cohort & Social Learning

In addition to the leaderboard, after each question, the course provides adaptive feedback, including references to how the broader cohort responded, anchoring individual responses in a collective context.

📊

Persistent Progress Bar

A visible progress indicator persists throughout the experience so learners always know where they stand — reducing anxiety and drop-off.

🌐

LMS-Free & SCORM-Free Delivery

The entire learning object runs as a self-contained HTML file hosted on a public website — no authoring tool, no SCORM package, no LMS required.

Technical Implementation

The development approach for this project included working collaboratively with an AI assistant to generate initial code structures, then I manually edited, customized, and iterated on the output ("vibe-coding"). This approach reflects a growing skill set for instructional designers who want to build custom learning experiences. This is particularly valuable in terms of reducing reliance on dedicated authoring platforms and learning management systems. Of course, custom experiences like this can also be embedded into traditional eLearning courses (Rise 360, etc.).

[Workflow] Describe feature to AI → Review generated code → Edit & customize → Test in browser → Iterate
[Stack] HTML5 · CSS3 · JavaScript · GitHub Pages (free hosting)
[Dependencies] Zero — no frameworks, no LMS, no authoring platform

Phase 1: Concept & Prompt Engineering

Before writing a line of code, I translated the ATD26 demo concepts into specific, actionable prompts for my AI collaborator. Describing the desired output and mechanics clearly produced far better initial drafts and reduced rework. To maximize efficiency (and minimize the use of AI tokens), I carefully scripted my AI prompts. This was particularly true with the initial prompt describing my requested output for the first draft. My pre-work included creating a template for the HTML code to upload along with my prompt.

Phase 2: AI-Generated Scaffolding

My AI partner produced the structural HTML skeleton, JavaScript logic for scoring and leaderboard sorting, and base CSS styling — giving me a working draft to iterate upon rather than a "blank page" starting point.

Phase 3: Human Customization & Refinement

During the iteration process, I edited the generated code directly while occasionally consulting AI as a technical advisor. Each round of editing deepened my understanding of the codebase. (And, although not my primary goal, helped develop my code-writing capabilities.)

Phase 4: Agile Iteration

Rather than treating any single version as final product, I published each iteration publicly and used the experience of interacting with the working product to drive the next round of improvements. As I reviewed each iteration, I continued to identify areas to improve as well as features I had not originally considered.

Agile Iterations: From POC to Product

For more than a decade, I have been a strong proponent of Agile methodologies. For this project especially, I embraced the concept of "shipping" early and iterating often. In showcasing this project, I thought it was important to demonstrate some early, less-impressive versions of the output. Because in the spirit of Agile, the final version was the direct result of the earlier POCs. Each version below essentially represents a sprint — a publishable artifact that demonstrated progress and informed the next build.

Sprint 1 → Functional Proof of Concept

Established the core mechanics: content slides, a three-question quiz, real-time scoring, and a basic leaderboard. Priority was proving the concept was technically achievable with no authoring tools. UI was minimal — function over form.

Sprint 2 → Social Layer & Cohort Identity

Added a name-entry field so learners appear on the leaderboard by name, introducing a true social dimension. Expanded to four questions and refined the progress and score display.

Sprint 3 → Slide-Based Flow & Final Polish

Restructured the experience into a true slide-by-slide flow mirroring a conventional eLearning course. Added prior-knowledge selection at entry, integrated the leaderboard into each question slide, and produced a more polished interface. Although functionality remained the primary objective, this iteration included some miminal aesthetic enhancements.

Key Challenges

The Hurdle: Replicating the leaderboard mechanic without a backend database. A "live" leaderboard typically requires a server to aggregate scores from multiple users — not available in a static HTML file.

The Fix: Used the browser's built-in memory (JavaScript arrays) to simulate multi-learner data within a single session, making the social comparison mechanic feel real in a local context. A clear next step for a production version would be adding a lightweight backend.

The Hurdle: Transitioning from a single-page scrolling layout (v1.0) to a true slide-based flow (v3.0) required rethinking how state — current slide, score, learner name — was managed across "screens" within a single HTML file.

The Fix: Worked with the AI to implement a JavaScript state machine pattern: slides are hidden and become visible based on the current state variable (similar to Storyline 360 triggers). Editing and understanding this logic firsthand gave me a much stronger mental model of how authoring tools like Articulate 360 manage slide transitions "under the hood".
